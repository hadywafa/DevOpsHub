# üìë **IAM Policies in AWS: Control Access with Precision**

> üìå "Who can do what, on which resource, under what condition?"

**IAM Policies** are the **brains behind access control** in AWS. They determine **who can do what**, **on which resources**, and **under what conditions**.

They‚Äôre written in **JSON**, evaluated by AWS every time an action is attempted, and are essential to implementing the **principle of least privilege**.

## üß† **What Are IAM Policies?**

IAM Policies are **permissions documents** attached to identities or resources. These JSON-based policies explicitly **allow** or **deny** actions on AWS services.

‚úÖ Think of a policy as a **"rules list"** saying:

> ‚ÄúUser X **can** read objects from this S3 bucket, but **can‚Äôt** delete them.‚Äù

---

## üß± **Key Components of a Policy**

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::mybucket/*"
    }
  ]
}
```

| üîπ Field    | üßæ Description                                                |
| ----------- | ------------------------------------------------------------- |
| `Version`   | Policy language version (always `"2012-10-17"` for AWS).      |
| `Statement` | One or more permission rules.                                 |
| `Effect`    | `"Allow"` or `"Deny"`. Deny always takes precedence.          |
| `Action`    | The operation to allow/deny (e.g., `"s3:GetObject"`).         |
| `Resource`  | The ARN(s) of the AWS resources to which this action applies. |

---

## **üîç Policy Evaluation Logic**

IAM policies use a **implicit default deny** model and evaluate requests in a specific way:

1. **üè† Default Deny**: By default, **all requests are implicit denied** until explicitly allowed by a policy.

2. **‚úÖ Explicit Allow**: If a policy explicitly allows an action, it will be granted even if a general **deny** is in place.

3. **‚ùå Explicit Deny**: If a policy explicitly denies an action, that denial **overrides any allows**. Deny always takes precedence over allow.

### ‚úçÔ∏è Example: Allow All S3 Actions on a Bucket

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::mybucket/*"
    }
  ]
}
```

### ‚úçÔ∏è Example: Deny All S3 Actions on a Bucket except for `"s3:GetObject"`

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Deny",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::mybucket/*"
    },
    {
      "Effect": "Allow",
      "Action": "*",
      "Resource": "arn:aws:s3:::mybucket/*"
    }
  ]
}
```

### üü¢ **1. Identity-Based Policy** (Most Common)

> üéØ What _can‚Äôt_ this **identity** (user, group, or role) do?

- ‚úÖ Attach to: **IAM users**, **groups**, **roles**
- üßæ Example:

```json
{
  "Effect": "Allow",
  "Action": "s3:ListBucket",
  "Resource": "arn:aws:s3:::my-bucket"
}
```

- üîÑ Can be:
  - **AWS Managed**
  - **Customer Managed**
  - **Inline**

üîê **Best For:**  
Controlling access for developers, EC2, Lambda, or service roles.

---

### üü• **2. Resource-Based Policy**

> üéØ Who _can‚Äôt_ do what on **this resource**?

- ‚úÖ Attach to: **S3 Buckets**, **Lambda**, **Secrets Manager**, etc.
- üßæ Example (S3 Bucket):

```json
{
  "Effect": "Allow",
  "Principal": { "AWS": "arn:aws:iam::111122223333:user/Alice" },
  "Action": "s3:GetObject",
  "Resource": "arn:aws:s3:::my-bucket/*"
}
```

- ‚úÖ Must include a `Principal` (i.e., who is allowed)

‚ö†Ô∏è **Important Note:**  
To allow **cross-account access**, a **resource-based policy** is required!

---

### üü§ **3. Role Trust Policy**

> üéØ Who _can assume_ this role?

- ‚úÖ Embedded in **IAM Role** definition
- üßæ Example:

```json
{
  "Effect": "Allow",
  "Principal": {
    "Service": "ec2.amazonaws.com"
  },
  "Action": "sts:AssumeRole"
}
```

üõ†Ô∏è **Use Cases:**

- Allow EC2 to assume a role (EC2 instance profile)
- Cross-account access
- AWS services using roles (Lambda, ECS)

---

### üîµ **4. Assumed-Role Session Policy**

> üéØ What _can‚Äôt_ be done **during a temporary session**?

- ‚úÖ Passed during `sts:AssumeRole` API call
- üßæ Example:

```json
{
  "Effect": "Allow",
  "Action": "s3:*",
  "Resource": "*"
}
```

- Only applies during the **session** lifespan

üõ†Ô∏è **Use Cases:**

- Limiting access for federated users
- Granting temporary limited permissions (e.g., consultants)

---

### üü£ **5. Permissions Boundary**

> üéØ What‚Äôs the **maximum** permissions this identity can get?

- ‚úÖ Attach to: IAM **users** or **roles**
- ‚úÖ Acts as a **guardrail**: Identity-based policies must be _within_ this boundary

üßæ Example:

```json
{
  "Effect": "Allow",
  "Action": "s3:*",
  "Resource": "*"
}
```

> If the boundary doesn't allow `s3:DeleteObject`, the user can't perform it, even if their identity policy allows it.

‚ö†Ô∏è Think of it as a **"maximum allowed scope"**.

---

### üü¶ **6. Service Control Policies (SCPs)**

> üéØ What‚Äôs the **maximum** any identity in this org can do?

- ‚úÖ Applies to: **AWS Organization Accounts**
- ‚úÖ Attached to: **OU**, **root**, or **accounts**
- ‚ùå SCPs don‚Äôt grant permissions ‚Äî they **limit** them.

üßæ Example: Deny deleting buckets

```json
{
  "Effect": "Deny",
  "Action": "s3:DeleteBucket",
  "Resource": "*"
}
```

üß† **Important:** Even if an IAM policy allows `s3:DeleteBucket`, SCP can block it.

## üéØ **Common Use Cases**

| Scenario                                 | Policy Type            | Description                                                         |
| ---------------------------------------- | ---------------------- | ------------------------------------------------------------------- |
| EC2 instance needs to access S3          | Identity-Based + Role  | Attach a policy to a role, and assign the role to the EC2 instance. |
| Share S3 bucket with another AWS account | Resource-Based         | Add a bucket policy with a cross-account `Principal`.               |
| Prevent S3 bucket deletion org-wide      | Service Control Policy | Use an SCP with `Deny: s3:DeleteBucket`.                            |
| Restrict session to read-only            | Session Policy         | Attach a read-only session policy when assuming a role.             |
| Define max permissions for developers    | Permissions Boundary   | Set a boundary limiting what policies can allow.                    |

## üí° **IAM Groups ‚â† Principals**

IAM **Groups** help you organize users, but they **can‚Äôt act on their own**.

- ‚úÖ You can attach policies to groups.
- ‚ùå You **can‚Äôt assign roles or trust policies** to groups.
