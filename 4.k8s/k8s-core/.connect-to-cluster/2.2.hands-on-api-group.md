# üß© **Kubernetes API Groups**

Kubernetes organizes its resources into **API groups** to support modularity, extensibility, and versioning. Each group contains related resources ‚Äî such as `pods`, `deployments`, or `configmaps` ‚Äî grouped by function or domain.

---

## üöÄ Why API Groups Matter

- **Modularity**: Clean separation of concerns (e.g., core vs RBAC vs batch workloads).
- **Extensibility**: New APIs can be added without disrupting core functionality.
- **Versioning**: Each group can evolve independently (e.g., `v1`, `v1beta1`).

---

## üîç How API Groups Are Structured

### 1. **Core API Group (`""`)**

- No explicit group name ‚Äî it‚Äôs the default.
- Contains foundational resources:
  - `Pods`, `Services`, `Nodes`, `Namespaces`, `ConfigMaps`, etc.
- These are critical for basic cluster operation.

### 2. **Named API Groups**

- Explicit group names used in the format: `<group>/<version>`.
- Examples:
  - `apps/v1` ‚Üí `Deployments`, `StatefulSets`, `DaemonSets`
  - `rbac.authorization.k8s.io/v1` ‚Üí `Roles`, `RoleBindings`, `ClusterRoles`
  - `batch/v1` ‚Üí `Jobs`, `CronJobs`
  - `networking.k8s.io/v1` ‚Üí `Ingress`, `NetworkPolicy`

### 3. **API Versions**

- Each group can have multiple versions:
  - `v1`, `v1beta1`, `v1alpha1`, etc.
- Newer versions may introduce breaking changes, enhancements, or deprecations.

## üìã Listing API Groups and Resources

Use `kubectl api-resources` to list all available resources and their metadata:

```bash
kubectl api-resources
```

### üßæ Sample Output

```plaintext
NAME           SHORTNAMES   APIGROUP                        NAMESPACED   KIND
pods           po           ""                              true         Pod
services       svc          ""                              true         Service
deployments    deploy       apps                            true         Deployment
jobs                        batch                           true         Job
cronjobs       cj           batch                           true         CronJob
roles                       rbac.authorization.k8s.io      true         Role
clusterroles                rbac.authorization.k8s.io      false        ClusterRole
```

---

## üß† Understanding the Output

| Column       | Meaning                                                    |
| ------------ | ---------------------------------------------------------- |
| `NAME`       | Resource name used in `kubectl` commands                   |
| `SHORTNAMES` | Abbreviations (e.g., `po` for `pods`, `cj` for `cronjobs`) |
| `APIGROUP`   | API group the resource belongs to (`""` means core group)  |
| `NAMESPACED` | `true` ‚Üí scoped to a namespace; `false` ‚Üí cluster-wide     |
| `KIND`       | The object type (e.g., `Pod`, `Deployment`, `Role`)        |

---

## üîé Filtering by API Group

To list resources from a specific group (e.g., `apps`):

```bash
kubectl api-resources --api-group=apps
```

### Example Output:

```plaintext
NAME          SHORTNAMES   APIGROUP   NAMESPACED   KIND
deployments   deploy       apps       true         Deployment
statefulsets  sts          apps       true         StatefulSet
daemonsets    ds           apps       true         DaemonSet
```

---

## üß™ Accessing Resources by Group

You can explicitly reference API groups in `kubectl` commands:

```bash
kubectl get deployments.apps
kubectl get cronjobs.batch
kubectl get roles.rbac.authorization.k8s.io
```

This is especially useful when:

- Multiple resources share the same name across groups
- You want to be precise in scripts or CI/CD pipelines

---

## üß≠ Real-World Use Case

Imagine you're auditing RBAC permissions:

```bash
kubectl get clusterroles.rbac.authorization.k8s.io
kubectl get rolebindings.rbac.authorization.k8s.io --namespace dev-team
```

Or inspecting workload controllers:

```bash
kubectl get deployments.apps
kubectl get jobs.batch
```

## üßµ Summary

- **Core API group (`""`)**: Essential resources like `Pods`, `Services`, `Nodes`.
- **Named API groups**: Logical separation (e.g., `apps`, `batch`, `rbac`).
- **Versioning**: Enables safe evolution of APIs.
- Use `kubectl api-resources` to explore and filter available resources.
