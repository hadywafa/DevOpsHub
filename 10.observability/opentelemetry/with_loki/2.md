Here’s the **exact step-by-step** to make `app.kubernetes.io/name` searchable in Loki as a **label** when you ingest logs via **OTel Collector → Loki OTLP HTTP (`otlphttp`)**.

---

## Step 0 — Confirm what you really have (Deployment vs Pod)

### ✅ The collector can only extract what exists on the **Pod**

Even if you “set it on the Deployment”, pods only get it if it’s on:

`Deployment.spec.template.metadata.labels`

Run:

```bash
kubectl get pod -n <ns> <pod-name> --show-labels | grep app.kubernetes.io/name
```

* If you see it → good, go to Step 1
* If you **don’t** see it → fix your Deployment (Step 0.1)

### Step 0.1 — Put it on the Pod template labels

In your Deployment:

```yaml
spec:
  template:
    metadata:
      labels:
        app.kubernetes.io/name: orders
```

Apply it, then re-check the pod labels.

---

## Step 1 — OTel Collector: extract this Pod label into a Resource Attribute

In your **logs collector** config, add/extend `k8sattributes`:

```yaml
processors:
  k8sattributes:
    auth_type: serviceAccount
    extract:
      labels:
        - from: pod
          key: app.kubernetes.io/name
          tag_name: app_kubernetes_io_name
```

Why `tag_name` like this?

* Loki doesn’t like `/` and `.` in label names, and it normalizes to `_` anyway.
* So we store it cleanly as `app_kubernetes_io_name`.

### Make sure your logs pipeline uses that processor

Example:

```yaml
service:
  pipelines:
    logs:
      receivers:  [filelog]          # or whatever you use
      processors: [k8sattributes, batch]
      exporters:  [otlphttp]
```

---

## Step 2 — OTel Collector: ensure you are exporting to Loki OTLP endpoint

Your exporter:

```yaml
exporters:
  otlphttp:
    endpoint: http://<loki-service>.<ns>.svc.cluster.local:3100/otlp
```

And in pipeline:

```yaml
service:
  pipelines:
    logs:
      exporters: [otlphttp]
```

---

## Step 3 — Loki: allow Structured Metadata (required for OTLP ingestion)

In Loki config:

```yaml
limits_config:
  allow_structured_metadata: true
```

If this is not enabled (or you’re on Loki < 3.0 with it off), Loki may reject payloads.

---

## Step 4 — Loki: promote your custom Resource Attribute to an **index label**

Now the key point:

After Step 1, Loki receives a **resource attribute** named:

`app_kubernetes_io_name`

But Loki will NOT index it as a label unless you tell it to.

### Option A (recommended): per-tenant mapping using `limits_config.otlp_config`

Add:

```yaml
limits_config:
  otlp_config:
    resource_attributes:
      attributes_config:
        - action: index_label
          attributes:
            - app_kubernetes_io_name
```

This says: “store this resource attribute as a Loki label”.

> If you want *only* your chosen labels (strict mode), you can also set `ignore_defaults: true`, but don’t do that unless you understand the impact (you might lose existing label-based queries).

---

## Step 5 — Restart / rollout

* Rollout restart the OTel Collector pods (logs collector)
* Rollout restart Loki (if config changed)

---

## Step 6 — Verify in Grafana (LogQL)

Query:

```logql
{app_kubernetes_io_name="orders"}
```

Also useful to sanity-check by starting broad:

```logql
{app_kubernetes_io_name=~".+"}
```

---

# Common failure reasons (so you don’t waste a day)

1. **Label is not on the Pod** (only on Deployment metadata)
2. `k8sattributes` not in the **logs pipeline** (only traces/metrics)
3. You extracted it, but didn’t **promote to index_label in Loki**, so it stayed only in structured metadata
4. You query using the original name `app.kubernetes.io/name` instead of normalized `app_kubernetes_io_name`
