# 1) What comes in an OTLP log record?

When the OpenTelemetry Collector sends a log to Loki (via `otlphttp`), the payload has 3 ÔÇťlayersÔÇŁ of attributes:

## A) Resource Attributes (WHO produced the logs?)

These describe the **source** of telemetry and are shared across many log lines.

Examples:

* `service.name`
* `k8s.namespace.name`
* `k8s.deployment.name`
* `k8s.pod.name`
* `cloud.region`

Ôťů Best fit for Loki **labels** because they define the stream/source.

---

## B) Scope Attributes (HOW was it produced?)

These come from the **instrumentation library / scope** (the thing producing telemetry).

Examples:

* instrumentation library name/version
* scope-level attributes (less common, but can exist)

Usually **not super useful for filtering dashboards**, so Loki often keeps them as structured metadata, or you may drop them.

---

## C) Log Attributes (WHAT is special about this log line?)

These are per log record (per event).

Examples:

* `http.method=GET`
* `user.id=123`
* `trace_id=...`
* `exception.stacktrace=...`

ÔÜá´ŞĆ These can explode in cardinality if you turn them into labels.

So Loki generally stores these as **structured metadata**, not index labels.

# 2) How Loki stores OTLP logs by default

When Loki receives OTLP logs:

## Ôťů LogLine

* `LogRecord.Body` becomes the **log text** (string)

## Ôťů Timestamp

* from `TimeUnixNano` or `ObservedTimestamp`

## Ôťů Index Labels (default selected resource attributes only)

By default Loki promotes a set of **Resource Attributes** into **labels**, like:

* `service.name`
* `k8s.namespace.name`
* `k8s.deployment.name`
* `k8s.pod.name` (they warn itÔÇÖs not recommended long-term)
  ÔÇŽand others you pasted.

Everything else becomes:

## Ôťů Structured Metadata

This includes:

* Resource attributes that were *not* promoted to labels
* Scope attributes
* Log attributes (except the body/timestamp fields)

**Important:** Structured metadata must be allowed:

```yaml
limits_config:
  allow_structured_metadata: true
```

If not, Loki rejects the payload.

# 3) The 3 actions Loki can take for attributes

Loki can apply these actions to attributes (based on type):

### Ôťů `index_label` (Resource Attributes only)

Promote to Loki label Ôćĺ **fast `{label="value"}` filtering**

* Only allowed for **Resource Attributes**
* Use this for **low-cardinality** stuff like `team`, `env`, `service.name`, `k8s.deployment.name`

### Ôťů `structured_metadata`

Store with each log line as structured metadata

* Allowed for **resource/scope/log attributes**
* Great for ÔÇťextra contextÔÇŁ that you donÔÇÖt want indexed

### Ôťů `drop`

Throw it away (save storage / avoid noise)

* Allowed for **resource/scope/log attributes**
* Use this for junk or high-volume attributes you never query

# 4) Why you should fear ÔÇťlabelsÔÇŁ in Loki (simple version)

Loki indexes by **label combinations**.
If you create too many unique combinations (high cardinality), Loki gets expensive/slower.

Ôťů Good labels (low cardinality):

* `env=prod|preprod`
* `team=billing|crm|payments`
* `service_name=orders|catalog`
* `k8s_namespace_name=...`

ÔÜá´ŞĆ Bad labels (high cardinality):

* `k8s_pod_name` (changes often)
* `service_instance_id` (unique per instance)
* `trace_id`, `request_id`, `user_id` (explodes)

ThatÔÇÖs why Loki docs warn about `k8s.pod.name` and `service.instance.id` as defaults.

---

# 5) ÔÇťDots become underscoresÔÇŁ and other transformations

Loki label names canÔÇÖt contain `.`, `/`, etc.

So Loki normalizes keys:

* `service.name` Ôćĺ `service_name`
* `k8s.namespace.name` Ôćĺ `k8s_namespace_name`

Also:

* nested attributes get flattened with `_`
* non-string values get stringified

So when you query, always use the **normalized** label/field name.

# 6) How to customize mapping (the 2 knobs)

You saw two config areas:

## A) `distributor.otlp_config.default_resource_attributes_as_index_labels`

This controls the **default allow-list** of *resource attributes* that become labels.

Example:

```yaml
distributor:
  otlp_config:
    default_resource_attributes_as_index_labels:
      - service.name
      - k8s.namespace.name
      - k8s.deployment.name
```

## B) `limits_config.otlp_config`

This is **per-tenant / override rules** to:

* ignore defaults
* add extra labels
* move attributes to structured metadata
* drop attributes

This is where `attributes_config` comes in.

# 7) The examples you pasted ÔÇö explained simply

## Example 1 (add one more label, keep defaults)

Ôťů Result:

* Everything else Ôćĺ structured metadata

## Example 2 (ONLY label one thing, ignore defaults)

> ÔÇťIgnore LokiÔÇÖs default label list. Only index `service.group`.ÔÇŁ

```yaml

```

Ôťů Result:

---

## Example 3 (fine-grained: label some, drop some scope/log attrs)

This one does 3 things:

1. Add `service.group` as label (keeping defaults)
2. Drop a specific scope attribute `method.name`
3. For log attributes:

   * keep `user.id` as structured metadata
   * drop all other log attributes

```yaml
limits_config:
  otlp_config:
    resource_attributes:
      attributes_config:
        - action: index_label
          regex: service.group

    scope_attributes:
      - action: drop
        attributes: [method.name]

    log_attributes:
      - action: structured_metadata
        attributes: [user.id]
      - action: drop
        regex: .*
```

Ôťů Result:

* Labels are controlled (resource attrs)
* Scope noise reduced
* Log attributes mostly removed except `user.id`

# 8) Practical recommendation for your Kubernetes case
