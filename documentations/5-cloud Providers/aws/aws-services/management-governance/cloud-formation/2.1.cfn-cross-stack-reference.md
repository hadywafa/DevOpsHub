# **üîó Cross-Stack References in AWS CloudFormation**

AWS CloudFormation **Cross-Stack References** allow multiple stacks to **share resources dynamically** without duplication.

‚úî **Improves infrastructure modularity** ‚Äì Split large stacks into smaller, manageable components.  
‚úî **Reduces duplication** ‚Äì Avoid redefining the same resources in every stack.  
‚úî **Ensures consistency** ‚Äì Prevents configuration mismatches between dependent stacks.

üîπ **Common Use Cases**  
‚úÖ Sharing **VPC IDs, Subnet IDs, Security Groups** across multiple stacks.  
‚úÖ Reusing a **common IAM role or S3 bucket** in different application stacks.  
‚úÖ Managing **networking infrastructure** separately from compute resources.

---

## **üõ†Ô∏è How Cross-Stack References Work?**

AWS CloudFormation enables one stack (**Producer Stack**) to **export** resource values, while another stack (**Consumer Stack**) can **import** those values.

**Key Components:**  
1Ô∏è‚É£ **Exports** ‚Äì Defined in the **Producer Stack**, making resource values available.  
2Ô∏è‚É£ **ImportValue** ‚Äì Used in the **Consumer Stack** to access exported values.

‚úÖ **Ensures multiple stacks can share infrastructure resources efficiently.**

---

## **üèóÔ∏è Implementing Cross-Stack References in CloudFormation**

### **üîπ 1Ô∏è‚É£ Creating the Producer Stack (Exporting Resources)**

The **Producer Stack** creates a resource (e.g., VPC) and **exports** its value using the `Outputs` section.

```yaml
AWSTemplateFormatVersion: "2010-09-09"
Description: "Producer Stack - Exports a VPC ID for use in other stacks"

Resources:
  MyVPC:
    Type: AWS::EC2::VPC
    Properties:
      CidrBlock: 10.0.0.0/16

Outputs:
  VPCID:
    Description: "Exports VPC ID for other stacks"
    Value: !Ref MyVPC
    Export:
      Name: MyApp-VPC-ID # Export name must be unique in the AWS region
```

‚úî **Defines an AWS resource (VPC) and makes it available for other stacks.**

---

### **üîπ 2Ô∏è‚É£ Creating the Consumer Stack (Importing Resources)**

The **Consumer Stack** uses `!ImportValue` to access the exported resource.

```yaml
AWSTemplateFormatVersion: "2010-09-09"
Description: "Consumer Stack - Uses an existing VPC from another stack"

Resources:
  MySubnet:
    Type: AWS::EC2::Subnet
    Properties:
      VpcId: !ImportValue MyApp-VPC-ID # Importing the exported VPC ID
      CidrBlock: 10.0.1.0/24
```

‚úî **References the `MyApp-VPC-ID` exported from the Producer Stack.**

### **3Ô∏è‚É£ Updating Cross-Stack References**

If a **Producer Stack modifies an exported resource**, AWS CloudFormation **prevents updates** if another stack is actively using the export.

üîπ **To update an export value:**  
1Ô∏è‚É£ **Delete the Consumer Stack first**  
2Ô∏è‚É£ **Update the Producer Stack**  
3Ô∏è‚É£ **Recreate the Consumer Stack with updated imports**

## **üîó Supported Functions in `Fn::ImportValue`**

You can use the following intrinsic functions inside `Fn::ImportValue`. However, these functions **cannot depend on a resource**:

| **Function**      | **Description**                                    |
| ----------------- | -------------------------------------------------- |
| **Fn::Base64**    | Encodes the imported value in Base64 format.       |
| **Fn::FindInMap** | Retrieves a value from a mapping table.            |
| **Fn::If**        | Returns a conditional value.                       |
| **Fn::Join**      | Concatenates imported values into a single string. |
| **Fn::Select**    | Picks a specific value from a list.                |
| **Fn::Sub**       | Replaces placeholders with values.                 |
| **Ref**           | References a parameter or logical resource ID.     |

> **üö® Important:**
>
> **You cannot use the short form (`!ImportValue`) with `!Sub`.**
>
> ‚ùå **Incorrect Usage:**
>
> ```yaml
> !ImportValue
>  !Sub '${NetworkStack}-SubnetID'
> ```
>
> ‚úÖ **Correct Usage:**
>
> ```yaml
> Fn::ImportValue: !Sub "${NetworkStack}-SubnetID"
> # or
> !ImportValue Fn::Sub: "${NetworkStack}-SubnetID"
> ```
>

## **‚ö†Ô∏è Limitations of Cross-Stack References**

| **Limitation**                             | **Description**                                                                |
| ------------------------------------------ | ------------------------------------------------------------------------------ |
| **Exports are region-scoped**              | Cross-stack references only work within the same AWS region.                   |
| **Exports cannot be updated**              | AWS does not allow modifying an exported value if another stack is using it.   |
| **Cannot export lists or complex objects** | Only **single values** (e.g., VPC ID, ARN, Security Group ID) can be exported. |
| **Export names must be unique**            | Each export name must be **unique per AWS region**.                            |

## **‚úÖ Alternative: Using AWS SSM Parameter Store for Cross-Stack References**

Instead of **Export/ImportValue**, AWS **Systems Manager Parameter Store (SSM)** allows cross-region and cross-account sharing.

### **1Ô∏è‚É£ Store a Value in SSM**

```yaml
Resources:
  StoreVpcID:
    Type: AWS::SSM::Parameter
    Properties:
      Name: "/Shared/VPCID"
      Type: String
      Value: !Ref MyVPC
```

‚úî Saves the **VPC ID** in SSM Parameter Store.

---

### **2Ô∏è‚É£ Retrieve Value in Another Stack**

```yaml
Parameters:
  VpcIdParam:
    Type: AWS::SSM::Parameter::Value<String>
    Default: "/Shared/VPCID"

Resources:
  MyInstance:
    Type: AWS::EC2::Instance
    Properties:
      SubnetId: !Ref VpcIdParam
```

‚úî Works across **AWS accounts and regions**.

## **üìú Best Practices for Cross-Stack References**

‚úÖ **Use `!ImportValue` only for static, foundational resources** (e.g., VPCs, IAM roles).  
‚úÖ **Avoid frequent updates** to exported values to prevent stack deletion.  
‚úÖ **Use AWS SSM Parameter Store** for **cross-region and cross-account sharing**.  
‚úÖ **Ensure Export names are unique** across the AWS region.  
‚úÖ **Keep stacks modular** ‚Äì Separate **networking, compute, and security** stacks.
